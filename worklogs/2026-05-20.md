# SOC Lab Worklog — 20.05.2026

**Focus:** VirusTotal integration · FIM active response pipeline · Custom detection rules

## What I Did
- Configured VirusTotal integration in ossec.conf — FIM events trigger hash lookup via free tier API
- Created remove-threat.sh active response script — auto-deletes files confirmed malicious by VirusTotal
- Tested full pipeline with EICAR standard test file dropped in /root
- Added /etc to realtime FIM monitoring scope
- Wrote two custom detection rules in local_rules.xml:
  - Rule 100001: /etc/shadow modification (T1003.008)
  - Rule 100002: executable dropped in /tmp (T1059)
- Verified both custom rules firing against real system events in Wazuh dashboard

## Alerts / Events Observed
- Rule 554 · Level 5 · File added to system — EICAR file detected by FIM in /root
- Rule 553 · Level 7 · File deleted — active response auto-deleted EICAR file
- Rule 87101 · Level 3 · VirusTotal: Error: Public API request rate limit reached — free tier 4 req/min exceeded during rapid testing
- Rule 100001 · Level 10 · FIM: /etc/shadow modified - possible credential access or manipulation · T1003.008
- Rule 100002 · Level 10 · FIM: Executable file dropped in /tmp - possible malware staging · T1059

## Key Finding
The detection-enrichment-response pipeline fired end-to-end: FIM detected the EICAR file in /root, triggered VirusTotal hash lookup, and active response deleted the file automatically without manual intervention. The free tier rate limit (4 req/min) caused a storm of 87101 errors during rapid testing but did not prevent active response from firing. This exposed a real production consideration — API throttling is required when FIM events are high-frequency. Both custom rules (100001 and 100002) verified firing against live system events with correct MITRE ATT&CK mappings.
